# Lessons Quick

Read this before every TARS work cycle. Keep it short; move detailed history to `docs/lessons-and-rules.md`.

## Core Rules

- Do not let important project knowledge live only in chat. Update `docs/decision-log.md`, `docs/lessons-and-rules.md`, or the relevant harness doc when a reusable decision emerges.
- For maintenance, root cause comes before edits. Guessing and patch stacking are forbidden.
- Preserve existing behavior unless the user explicitly asks to change it.
- Keep fixes small. Do not turn a correction into a redesign or broad refactor.
- Verification is evidence. Do not claim completion without running checks or clearly reporting why checks could not run.
- Promote rules gradually: repo docs first, project-local skill next, global skill only after repeated real use.
